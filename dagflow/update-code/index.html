    Code Location

    Dagster separates the control plane from user code. See the Dagster code locations architecture for details.

    To update a code location, build a new source.pex and deps.pex, upload them to the app-managed PEX stage, and point the service comment at the new version with deploy_code_location.

    Build PEX Artifacts

    The runtime expects two files per code-location version:

    • source.pex: your Dagster definitions and project code.
    • deps.pex: the project's runtime distributions, including Dagster and its PostgreSQL storage integration. The runtime image supplies Dagflow's compute-log manager without dependencies.

    Example project layout:

    .
|-- requirements.txt
`-- finance
    |-- __init__.py
    `-- definitions.py

    Example requirements.txt:

    dagster==1.12.1
dagster-postgres==0.28.1
snowflake-connector-python>=3.15,<4

    dagster and dagster-postgres must be direct project requirements. This makes the selected deps.pex the only Dagster environment used by both the code server and its run workers. The provider-owned image supplies Dagflow's runtime extension without dependencies. Use the dagster-postgres companion release compatible with the selected Dagster version. Optional and development dependency groups are not bundled automatically.

    The builder combines base dependencies declared in requirements.txt, setup.py, and pyproject.toml. Relative local packages are built into the source artifact recursively. Virtual environments, VCS metadata, caches, and build output are excluded from the filtered working source.

    Use a stable version string when staging PEX artifacts:

    @dagflow_app.core.pex/finance/2026-06-01/source.pex
@dagflow_app.core.pex/finance/2026-06-01/deps.pex

    Deploy with Snowflake CLI

    Install the Dagflow CLI plugin in the same environment as Snowflake CLI:

    pip install dagflow-cli-plugin

    Enable the external plugin in your Snowflake CLI config:

    [cli.plugins.dg]
enabled = true

    Then use your existing Snowflake CLI connection to build, upload, and deploy. For a Dagster dg project, Dagflow infers the code-location name and target module from [tool.dg]:

    snow dg deploy \
  --app-name dagflow_app \
  --path . \
  --version 2026-06-01 \
  --blocking

    For a non-dg project, pass the code-location name and one explicit entrypoint:

    snow dg deploy finance \
  --app-name dagflow_app \
  --path . \
  --version 2026-06-01 \
  --module-name finance.definitions \
  --blocking

    The plugin builds source.pex and deps.pex in a temporary directory, uploads them to @dagflow_app.core.pex/finance/2026-06-01/, and calls the Dagflow server code-location API. Use code-location package when you want to keep the generated files locally for inspection or reuse. Use --python-version 3.12 when deploying to the Python 3.12 runtime. If the requested version differs from the active CLI Python, a matching interpreter such as python3.12 must be on PATH.

    If you have already built PEX artifacts, use code-location add to upload and register them without packaging the project again:

    snow dg code-location add finance \
  --app-name dagflow_app \
  --version 2026-06-01 \
  --module-name finance.definitions \
  --source-pex dist/dagflow/finance/2026-06-01/source.pex \
  --deps-pex dist/dagflow/finance/2026-06-01/deps.pex \
  --blocking

    Launch a job with Dagster-style job launch naming:

    snow dg job launch \
  --app-name dagflow_app \
  --location finance \
  --job my_job \
  --config run-config.yaml \
  --tag snowflake/cpu=500m

    You can also build artifacts without deploying:

    snow dg code-location package . \
  --location-name finance \
  --version 2026-06-01 \
  --module-name finance.definitions

    code-location package writes local artifacts under dist/dagflow/<location>/<version>/ unless you pass --output-dir.

    Update an Existing Code Location

    An APP_EDITOR can update code by deploying a new PEX version:

    snow dg code-location add finance \
  --app-name dagflow_app \
  --version 2026-06-01 \
  --module-name finance.definitions \
  --source-pex dist/dagflow/finance/2026-06-01/source.pex \
  --deps-pex dist/dagflow/finance/2026-06-01/deps.pex \
  --blocking

    Use --no-blocking when a release script should return after requesting the service update instead of waiting for the new code server to become ready.

    Refresh Dagster

    Updating an existing code location does not require reinstalling Dagflow. The Dagflow server API updates the service comment, then reloads the code location in Dagster. That calls the PEX host's ReloadCode gRPC method, which activates the latest requested version before Dagster refreshes metadata.

    Adding or removing a code location changes the Snowflake service list. The webserver and daemon refresh their workspace from running services in the code-location schema, so code-location API calls do not restart core.dagster.

    Reload one deployed location from the CLI:

    snow dg code-location reload finance --app-name dagflow_app

    Reload the full Dagster workspace after out-of-band service changes:

    snow dg code-location reload-all --app-name dagflow_app

    Restart the control plane manually when needed:

    CALL dagflow_app.app_public.restart_dagster();

    Remove a Code Location

    Remove code locations that should no longer appear in Dagster:

    snow dg code-location delete finance --app-name dagflow_app

    Dagflow drops the code-location service through the server API. The webserver and daemon stop listing it after workspace refresh.

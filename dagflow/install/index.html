    Installation

    Prerequisites

    You need a Snowflake role that can install the Dagflow Native App and grant its requested privileges. The app needs account-level privileges to create compute pools and bind service endpoints.

    You also need PEX artifacts for each Dagster code location: source.pex for project code and deps.pex for third-party dependencies.

    Install the App

    If you install from the Snowflake Marketplace UI, follow the UI prompts and skip this section. To run the same steps manually, continue below.

    After installation, grant requested privileges if they were not auto-granted:

    GRANT CREATE COMPUTE POOL ON ACCOUNT TO APPLICATION dagflow_app;
GRANT BIND SERVICE ENDPOINT ON ACCOUNT TO APPLICATION dagflow_app;

    Snowflake invokes the app grant callback after required privileges are granted. To retry startup manually:

    CALL dagflow_app.app_public.start_app();
CALL dagflow_app.app_public.service_status();

    Grant Dagflow application roles to users or roles that need the Dagster UI:

    GRANT APPLICATION ROLE dagflow_app.APP_VIEWER TO ROLE <viewer_role>;
GRANT APPLICATION ROLE dagflow_app.APP_LAUNCHER TO ROLE <launcher_role>;
GRANT APPLICATION ROLE dagflow_app.APP_EDITOR TO ROLE <editor_role>;
GRANT APPLICATION ROLE dagflow_app.APP_ADMIN TO ROLE <admin_role>;

    Use the least powerful role that fits each user. See RBAC for allowed actions.

    Prepare PEX Artifacts

    Build your Dagster project into source.pex and deps.pex, then upload those files to the app-managed PEX stage with AUTO_COMPRESS=FALSE.

    Add a Code Location

    Upload and deploy the code-location version:

    snow dg code-location add finance \
  --app-name dagflow_app \
  --version 2026-05-29 \
  --module-name finance.definitions \
  --source-pex source.pex \
  --deps-pex deps.pex \
  --blocking

    Dagflow creates an app-owned code location named FINANCE with the provider-owned runtime image and records the desired PEX version in the service comment.

    Pass --no-blocking when automation should return after requesting the service update instead of waiting for readiness.

    After the service is ready, open the Dagster UI from the Snowflake service endpoint for core.dagster.

    Compute Resources

    Dagflow creates three SPCS compute pools in the consumer account:

    Compute pool Used for Instance family Nodes
    <app_name>_CONTROL_POOL Dagster UI, daemon, and storage GEN_ARM_G1_8 1
    <app_name>_CODE_LOCATION_POOL Registered code locations GEN_ARM_G1_4 1-20
    <app_name>_RUN_POOL Pipeline run workers GEN_ARM_G1_8 1-50

    Run workers default to 0.25 vCPU and 512Mi memory per run. Users can request different resources with snowflake/cpu and snowflake/memory tags.

    PEX Code Deployment Plan

    This plan replaces user-provided Docker images for code locations with provider-owned Dagflow runtime services that load user code from PEX artifacts.

    Goals

    • Keep all SPCS services inside the Native App.
    • Use only provider-known runtime images in services.
    • Let users deploy code without pushing a service image that the app manifest cannot know ahead of time.
    • Make code updates cheap: upload new PEX files and update service metadata.
    • Preserve the Dagster code-location model, where each location has an independently reloadable user-code server.

    Runtime Model

    Each Dagster code location maps to one SPCS service inside the application:

    app.code_locations.<location_name>

    The service name is the code-location name. The service image is always a Dagflow-owned runtime image:

    service: app.code_locations.finance
image: provider/dagflow-code-location-runtime-py311:<runtime-version>
command: dagflow-pex-host
comment: desired PEX version and entrypoint metadata

    The service runs a small PEX host process. The PEX host reads the service comment, derives artifact paths from the service name and version, and starts a Dagster gRPC subprocess from the staged PEX files.

    The code-location runtime image is separate from the Dagflow control-plane image. Dagflow publishes Python-versioned runtime variants, such as dagflow-code-location-runtime-py311 and dagflow-code-location-runtime-py312, so code locations and run workers can use a different Python runtime than the webserver/daemon.

    Artifact Layout

    Use deterministic paths on a Dagflow-managed internal stage:

    @CORE.PEX/<location_name>/<version>/source.pex
@CORE.PEX/<location_name>/<version>/deps.pex

    The runtime contract should use stable filenames, not wildcard lookup. The service name supplies location_name; the service comment supplies version.

    Example:

    @CORE.PEX/finance/abc123/source.pex
@CORE.PEX/finance/abc123/deps.pex

    Service Comment Contract

    The service comment is the desired-state pointer for the service. Keep it small:

    {
  "version": "abc123",
  "entrypoint": {
    "type": "module",
    "name": "finance.definitions"
  }
}

    Supported entrypoint forms:

    {"type": "module", "name": "finance.definitions"}
{"type": "package", "name": "finance"}
{"type": "python_file", "path": "definitions.py"}

    The PEX host derives:

    location_name = current service name
version = comment.version
source_pex = @CORE.PEX/{location_name}/{version}/source.pex
deps_pex = @CORE.PEX/{location_name}/{version}/deps.pex

    Deploy Flow

    Client side:

    1. Build source.pex and deps.pex.
2. Upload them to @CORE.PEX/<location>/<version>/.
3. Call the Dagflow server code-location API with location, version, and entrypoint.

    Server side:

    PUT /code-locations/{location}
  -> validate source.pex and deps.pex exist on stage
  -> create the code-location service if it does not exist
  -> otherwise ALTER SERVICE SET COMMENT to the new desired metadata

    Adding a location creates a service from the Dagflow runtime image:

    CREATE SERVICE app.code_locations.finance
  IN COMPUTE POOL ...
  FROM SPECIFICATION ...
  COMMENT = '{"version":"abc123","entrypoint":{"type":"module","name":"finance.definitions"}}';

    Updating a location does not create or replace the service. It only updates the comment:

    ALTER SERVICE app.code_locations.finance
SET COMMENT = '{"version":"def456","entrypoint":{"type":"module","name":"finance.definitions"}}';

    PEX Host Reconciliation

    Each code-location service owns the public Dagster gRPC port. The PEX host intercepts Dagster's ReloadCode RPC and proxies every other Dagster gRPC method to the active user-code subprocess on an internal port.

    Initial service startup and each ReloadCode request reconcile desired state:

    1. Read own service comment.
2. Parse desired version and entrypoint.
3. If desired version equals active version, do nothing.
4. Copy source.pex and deps.pex from the derived stage paths.
5. Store the PEX files in a versioned local directory.
6. Start a new subprocess on an internal port:
   source.pex -m dagster api grpc ...
7. Health-check the new Dagster gRPC subprocess.
8. Re-read the service comment before activation.
9. If the comment still requests the same version, switch active subprocess.
10. Stop the old subprocess.

    On service restart, the PEX host reads the service comment and reconstructs the active subprocess from the staged artifacts. After deploy_code_location updates the service comment, activation is triggered by Dagster's normal reloadRepositoryLocation flow, which calls the code location's ReloadCode gRPC method.

    Concurrency Model

    The service comment is authoritative desired state. Concurrent deploys are last-write-wins:

    deploy v1 -> ALTER SERVICE SET COMMENT = v1
deploy v2 -> ALTER SERVICE SET COMMENT = v2
reloadRepositoryLocation -> ReloadCode -> PEX host reads v2 -> loads v2

    No durable deploy lock is required for correctness under this model. The PEX host must avoid committing stale work:

    1. Host sees desired v1 and starts preparing v1.
2. Another deploy changes the comment to v2.
3. Before activating v1, host reads the comment again.
4. Since desired is now v2, host discards v1 and reconciles v2.

    An in-memory per-service lock is still useful inside the PEX host to avoid duplicate local reload work, but correctness comes from always reconciling to the latest service comment.

    Status

    The service comment records desired state only. Runtime status should come from the PEX host:

    {
  "desired_version": "def456",
  "active_version": "def456",
  "status": "ACTIVE",
  "error": null
}

    The first implementation can expose status through logs and an internal status endpoint. A later version can persist status if the UI or SQL APIs need durable history.

    RBAC

    APP_EDITOR should be allowed to:

    • add a code location
    • remove a code location
    • deploy or update a code location
    • refresh the Dagster workspace

    APP_VIEWER can view state but cannot deploy.

    The Native App owns service creation inside the application. Users deploy code by uploading PEX artifacts and updating service desired state, not by providing runtime service images.

    Why This Fits Native Apps

    The old image-based model requires creating services from user-pushed images. That fails when the image is not in the current application version manifest and cannot be known by the provider ahead of time.

    The PEX model keeps service images fixed and manifest-known:

    provider image -> stable SPCS service
user code -> staged PEX files
deploy -> ALTER SERVICE SET COMMENT

    This uses Snowflake service metadata as the desired-state signal and avoids creating runtime services in an app-created external database.
